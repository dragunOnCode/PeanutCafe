# Lobster 前端需求设计（多 Agent 协作聊天室）

## 1. 文档目的

本文档用于指导前端实现一个“多 Agent 协作聊天室”，兼顾产品体验与调试能力。  
需求参考 Cat Cafe 的功能演示，并结合当前 Lobster 后端已落地的 WebSocket 事件模型进行分阶段设计。

## 2. 产品目标

### 2.1 核心目标

- 在一个会话中与多个 Agent 协作，无需在多窗口来回切换
- 支持 `@Agent` 精确召唤和多 Agent 并行响应
- 可观测 Agent 工作过程（思考、流式输出、完成、跳过、报错）
- 保留调试能力（原始事件、链路日志、异常定位）

### 2.2 实施目标

- 第一阶段可上线：完整聊天闭环 + 事件可视化 + 流式渲染
- 第二阶段增强：协作链路、状态总览、导出与会话管理
- 第三阶段扩展：A2A 可视化、会话链、富文本与移动端适配

## 3. 用户角色

- 单人维护者（你）：既要快速开发，也要快速定位联调问题
- 开发/测试协作者：验证复杂协作链路是否符合预期
- 观察者角色（可选）：只读查看会话与日志

## 4. 术语定义

- Agent：如 Claude/Codex/Gemini 的独立智能体
- Mention：`@AgentName` 触发指定 Agent 参与
- A2A：Agent-to-Agent，Agent 在回复中继续召唤其他 Agent
- Session：一次持续会话的上下文边界
- Thread：会话集合的业务话题容器（后续阶段）

## 5. 功能需求（FR）

### FR-1 多 Agent 聊天基础能力（P0）

- 建立 `/chat` 连接并完成鉴权与会话加入
- 支持发送普通消息和 `@Agent` 消息
- 支持一次 @ 多个 Agent 并行响应
- 消息区按时间线展示：用户消息、Agent 消息、系统消息
- 消息显示身份信息：`agentName`、`role`、`createdAt`

验收：
- 能稳定收发 `message:send`、`message:received`、`chat:history`
- 在一条消息中 `@Claude @Codex` 可触发双 Agent 处理

### FR-2 Agent 状态与协作可视化（P0）

- 展示 Agent 当前状态：`thinking` / `responding(streaming)` / `idle` / `error` / `skip`
- 在消息流中显示 Agent 决策原因（来自 `agent:thinking`、`agent:skip`）
- 支持顶部状态栏，按 Agent 卡片展示最近状态和更新时间

验收：
- `agent:thinking` 到达后，状态栏在 100ms 内刷新
- `agent:error` 时状态卡片与消息区都有告警信息

### FR-3 流式响应渲染（P0）

- 监听 `agent:stream` 并按 `sessionId + agentId` 聚合
- 实时更新正在输出的消息气泡
- `agent:stream:end` 到达后将草稿固化为最终消息
- 支持切换“流式实时视图 / 最终文本视图”

验收：
- 流式内容不乱序、不串 Agent
- 结束事件后不再继续追加 delta

### FR-4 事件调试控制台（P0）

- 记录全部入站/出站事件（含原始 payload）
- 支持事件过滤：事件名、Agent、时间范围、关键词
- 支持导出当前会话日志为 JSON
- 提供“复制为问题报告片段”的快捷操作

验收：
- 事件日志与聊天 UI 可以互相定位（点击消息可跳到事件）
- 1000 条事件内筛选与滚动保持流畅

### FR-5 会话与线程管理（P1）

- 支持会话切换、重命名、归档、删除
- 会话列表按最近活跃排序
- 支持从历史会话恢复上下文
- 支持导出会话 Markdown（基础文本版）

验收：
- 切换会话时自动断开旧连接并建立新连接
- 历史会话加载可控（分页或最近 N 条）

### FR-6 A2A 协作链路展示（P1）

- 当消息中出现 Agent 对 Agent 调用时，前端以“协作链”展示
- 展示链路结构：`A -> B -> C`（时间顺序）
- 标记公开输出与内部过程（默认只显示公开输出）

验收：
- 用户能识别本轮响应是“用户触发”还是“A2A 触发”
- 链路面板可追溯每个节点的触发消息

### FR-7 配置与个性化（P2）

- Agent 配置面板（前端展示层）：模型标签、预算、工具权限（只读或可编辑，取决于后端接口）
- 输入框支持模板和快速命令（如常用 @ 组合）
- 主题模式与布局偏好存储（本地）

### FR-8 富内容与附件（P2）

- 支持 Markdown、代码块高亮、差异对比块（Diff Block）
- 预留图片/附件消息显示位
- 富内容导出（Markdown/图片）按阶段实现

## 6. 后端事件映射要求

前端必须完整消费以下事件（以当前文档为准）：

- 连接与会话：`connection:error`、`chat:history`、`session:presence`、`user:joined`、`user:left`
- 消息：`message:received`、`message:error`、`message:mention`
- Agent：`agent:thinking`、`agent:skip`、`agent:response`、`agent:error`
- 流式：`agent:stream`、`agent:stream:end`
- 系统：`system:notification`、`exception`

客户端发送事件：

- `message:send`（必选）
- `session:join`、`agent:stop`（按后端实现进度接入）

## 7. 页面信息架构

### 7.1 页面清单

- 登录/连接页
- 聊天主页面（默认）
- 会话管理抽屉（P1）
- 协作链路面板（P1）
- 调试控制台面板（P0，可抽屉形式）

### 7.2 聊天主页面布局

- 左：会话列表 + Agent 状态栏
- 中：消息流 + 输入区
- 右：事件调试面板（可收起）

## 8. 状态模型设计

- `ConnectionState`：连接状态、重连次数、错误原因、时延估计
- `AgentStateMap`：每个 Agent 的状态、最近事件、最后活动时间
- `ConversationState`：消息列表、stream 草稿、历史分页信息
- `SessionState`：当前 session、session 列表、成员统计
- `DebugEventState`：事件索引、过滤条件、导出快照

## 9. 非功能需求（NFR）

- 性能：消息发送到 UI 可见延迟 < 100ms（本地网络）
- 稳定性：断线重连后不重复订阅、不丢最终消息
- 可维护性：事件类型定义集中在 `shared/types/events.ts`
- 安全性：JWT 默认仅内存保存；敏感日志脱敏后导出
- 可测试性：核心状态机与 stream 合并逻辑必须有单测

## 10. 里程碑与范围控制

### M1（P0，先可用）

- 连接鉴权、消息收发、@提及
- Agent 状态栏、流式渲染
- 调试控制台与 JSON 导出

### M2（P1，协作增强）

- 会话管理（列表/切换/归档）
- A2A 协作链路面板
- Markdown 导出与基础统计

### M3（P2，产品化）

- Agent 配置中心
- 富文本组件（卡片、Diff、清单）
- 移动端适配与 PWA（可选）

## 11. 验收标准（版本级）

- V1（M1）：
  - 完成多 Agent 聊天闭环并支持流式显示
  - 事件与消息可双向追踪
  - 异常可见且可导出复盘
- V2（M2）：
  - 会话与协作链路可视化
  - 具备基础对话管理能力
