# 系统核心特性说明

> 多Agent群聊式协作编码系统的核心特性与设计理念

---

## 🎯 核心设计理念

### 1. 群聊式交互，而非机械协作

**传统方式**（流水线）：
```
用户 → Agent1 → Agent2 → Agent3 → 结果
```
❌ 问题：僵化、缺乏灵活性、Agent无法互动

**Lobster方式**（群聊）：
```
┌──────────────────────────────┐
│     聊天室 (Chatroom)        │
│                              │
│  👤 用户                     │
│  🤖 Claude (在线)            │
│  🤖 Codex (在线)             │
│  🤖 Gemini (在线)            │
│                              │
│  大家共享工作空间和记忆       │
└──────────────────────────────┘
```
✅ 优势：自然、灵活、Agent可以互相对话

## 🌟 核心特性

### 特性1：@提及召唤

**功能描述**：用户使用 `@Agent名称` 直接召唤特定Agent。

**使用示例**：

```
用户: @Claude 帮我设计一个用户系统
👉 Claude必定响应

用户: @Codex @Gemini 你们看看这个设计怎么样
👉 Codex和Gemini都会响应

用户: @Claude 根据Gemini的建议修改一下
👉 Claude会参考Gemini之前的消息（共享记忆）
```

**技术实现**：
- 正则表达式解析：`/@(\w+)/g`
- 被@的Agent收到 `Priority: HIGH` 标记
- 必定触发响应（除非Agent离线）

---

### 特性2：Agent主动感知与决策

**功能描述**：Agent像真实团队成员一样，主动感知聊天室消息并决定是否参与。

**决策机制**：

```typescript
// 每条新消息广播后，所有Agent并行决策
for (const agent of allAgents) {
  const decision = await agent.shouldRespond(message, context);
  
  if (decision.should) {
    // 通知前端："Claude 决定响应（原因：涉及架构设计）"
    // 开始生成响应
  }
}
```

**决策规则（三级）**：

| 级别 | 触发条件 | 优先级 | 示例 |
|------|---------|--------|------|
| **Level 1** | 被@提及 | 🔴 必定响应 | `@Claude` → Claude响应 |
| **Level 2** | 关键词匹配 | 🟡 很可能响应 | "设计"→Gemini响应 |
| **Level 3** | 上下文相关 | 🟢 可能响应 | 对话涉及UI→Gemini可能参与 |

**行为示例**：

```
用户: 我想做一个电商网站
[系统]: Claude 决定响应（消息涉及系统设计）
[系统]: Gemini 决定响应（我可以提供设计建议）

Claude: 我建议使用微服务架构...
Gemini: 电商网站的首页设计应该...

用户: 继续
Claude: 好的，我来实现购物车模块...

[5秒后，Codex检测到新代码]
Codex: 我看到Claude生成了购物车代码，让我审查一下...
       代码逻辑没问题，建议添加库存检查...
```

---

### 特性3：共享工作空间

**功能描述**：所有Agent和用户共享同一个工作空间，包含代码、文档、任务等。

**工作空间内容**：

```
Workspace (会话ID: abc-123)
├── 📁 代码仓库
│   ├── UserController.ts (Claude创建, v2)
│   ├── AuthService.ts (Claude创建, v1)
│   └── LoginPage.vue (Gemini建议, Claude实现, v1)
├── 📋 任务列表
│   ├── ✅ 设计用户表结构 (Claude, 已完成)
│   ├── 🔄 实现JWT认证 (Claude, 进行中)
│   └── ⏸️ 添加登录界面 (Gemini, 待处理)
├── 📚 技术文档
│   ├── 架构设计文档 (Claude)
│   └── UI设计规范 (Gemini)
└── 🎯 决策记录
    └── 使用JWT而非Session (Claude提议, Codex同意)
```

**实时同步**：
- ✅ 任何Agent修改工作空间 → 立即同步到Redis
- ✅ 其他Agent通过Redis Pub/Sub实时感知变化
- ✅ 前端通过WebSocket实时更新界面

**版本追踪**：

```typescript
// 文件版本历史
{
  path: 'UserController.ts',
  versions: [
    { version: 1, author: 'Claude', createdAt: '2026-02-18 10:00' },
    { version: 2, author: 'Claude', updatedAt: '2026-02-18 10:05', note: '根据Codex建议添加验证' },
    { version: 3, author: 'Claude', updatedAt: '2026-02-18 10:10', note: '根据Gemini建议优化错误提示' }
  ]
}
```

### 特性4：共享记忆，无需重复上下文

**功能描述**：所有Agent共享同一套记忆系统，新加入的Agent也能立即了解上下文。

**记忆层次**：

```
┌────────────────────────────────────┐
│      三层记忆架构                   │
├────────────────────────────────────┤
│                                    │
│  🧠 短期记忆 (5分钟)               │
│     最近20条消息                   │
│     所有Agent共享                  │
│     ↓                              │
│  💾 长期记忆 (永久)                │
│     完整对话历史                   │
│     向量化索引                     │
│     语义检索                       │
│     ↓                              │
│  🤝 共享记忆 (实时)                │
│     任务状态                       │
│     代码仓库                       │
│     技术决策                       │
│     话题追踪                       │
│                                    │
└────────────────────────────────────┘
```

**使用场景**：

```
用户: 我们用什么数据库？
Claude: PostgreSQL

[10分钟后]

用户: @Codex 写一个数据库查询
Codex: 好的，使用PostgreSQL...
       👆 Codex自动从共享记忆中获取了"用户选择PostgreSQL"

[无需用户重复说明数据库类型！]
```

**记忆包含的内容**：

| 内容类型 | 存储位置 | 示例 |
|---------|---------|------|
| 对话历史 | Redis + PostgreSQL | 所有消息记录 |
| 用户目标 | SharedMemory | "构建电商网站" |
| 技术决策 | SharedMemory | "使用NestJS+PostgreSQL" |
| 代码文件 | Workspace | UserController.ts |
| 任务进度 | Workspace | "登录功能 80%完成" |
| 待解决问题 | SharedMemory | "如何处理高并发？" |

### 特性5：双模式切换（自由 ↔ 结构化）

**自由对话模式**（默认）：
- Agent像人类一样自由对话
- 无固定流程
- 支持多Agent同时响应
- 灵活、自然、适合探索性任务

**结构化工作流模式**（可选）：
- Agent按预定流程执行
- 固定顺序：Claude → Codex → Gemini
- 适合明确的开发任务
- 确保完整性

**切换方式**：

```
用户输入：
"/workflow start"    → 切换到结构化模式
"/free"              → 切换回自由模式

或者：
用户: 使用完整开发流程
[系统]: 已切换到工作流模式
```

**混合使用**：

即使在工作流模式下，Agent仍可响应用户的即时提问：

```
[工作流运行中...]
Claude: 正在生成代码...（30%）

用户: @Gemini 你觉得这个功能需要什么样的界面？
Gemini: 我建议使用卡片式布局...（立即响应，不等工作流）

[工作流继续...]
Claude: 代码生成完成（100%）
```

### 特性6：Agent个性与专业分工

每个Agent有明确的角色定位和个性：

| Agent | 角色 | 性格 | 响应风格 |
|-------|------|------|---------|
| **Claude** | 架构师 + 全栈工程师 | 专业、严谨 | 先分析→设计→实现→说明 |
| **Codex** | 代码审查员 | 批判性、注重细节 | 发现问题→分析影响→给建议 |
| **Gemini** | 设计师 + 产品经理 | 创意、友好 | 理解需求→提方案→优化体验 |

**实际对话体现**：

```
用户: 实现用户登录

Claude: 
我来设计登录系统的架构。首先，我们需要：
1. 数据库Schema（User表）
2. JWT认证中间件
3. Controller和Service层
[生成完整代码...]

Gemini:
从用户体验角度，我建议：
1. 添加"记住我"功能
2. 支持第三方登录（Google/GitHub）
3. 登录页面使用渐变背景+卡片式设计
[提供设计方案...]

Codex:
我审查了Claude的代码，整体设计良好。几点建议：
1. 第15行缺少密码强度验证
2. 建议添加登录失败次数限制（防暴力破解）
3. Token应该存储在HttpOnly Cookie中
[给出修改建议...]
```

## 🔥 系统亮点

### 1. 智能决策，减少噪音

Agent不会对每条消息都响应，避免聊天室过于嘈杂：

```
用户: 今天天气真好
[所有Agent: 不响应，与编码无关]

用户: 我想优化一下代码性能
[Claude: 响应，涉及优化]
[Codex: 响应，涉及代码质量]
[Gemini: 不响应，与设计无关]
```

### 2. 上下文共享，高效协作

Agent不需要重复询问基础信息：

```
❌ 传统方式：
用户 → Claude: 用什么数据库？
用户回答: PostgreSQL
用户 → Codex: 写个查询（需要再说一次PostgreSQL）
用户 → Gemini: 设计界面（需要再说一次需求）

✅ Lobster方式：
用户: 用PostgreSQL
Claude: 好的，我来设计Schema...
[30秒后]
Codex: 我来写个查询... (自动知道是PostgreSQL)
Gemini: 数据展示界面... (自动知道需求背景)
```

### 3. 代码版本追踪

所有代码修改都有明确记录：

```
UserController.ts 历史：
- v1: Claude创建 (2026-02-18 10:00)
- v2: Claude修改 (10:05) - 基于Codex建议：添加输入验证
- v3: Claude修改 (10:10) - 基于Gemini建议：优化错误提示
```

### 4. 灵活可扩展

**添加新Agent只需3步**：
1. 编辑 `agents.config.json`
2. （可选）实现自定义Adapter
3. 重启应用

**示例 - 添加GPT-4**：

```json
{
  "id": "gpt4-001",
  "name": "GPT4",
  "type": "custom",
  "role": "通用AI助手",
  "callType": "http",
  "config": {
    "apiUrl": "https://api.openai.com/v1/chat/completions"
  },
  "enabled": true
}
```

完成！现在聊天室有4个Agent了。

## 🚀 技术优势

### 1. 性能优化

- ✅ **并行决策**：所有Agent同时判断，不阻塞
- ✅ **流式响应**：用户实时看到生成过程
- ✅ **缓存共享记忆**：Redis缓存，毫秒级读取
- ✅ **异步处理**：Agent生成不阻塞消息处理

### 2. 可靠性保证

- ✅ **自动重试**：Agent调用失败自动重试3次
- ✅ **降级策略**：某Agent离线，其他Agent继续工作
- ✅ **健康检查**：每30秒检测Agent状态
- ✅ **持久化存储**：所有消息永久保存

### 3. 扩展性设计

- ✅ **动态Agent管理**：运行时添加/移除Agent
- ✅ **插件化架构**：新Agent只需实现ILLMAdapter接口
- ✅ **水平扩展**：支持多实例部署（Redis共享状态）
- ✅ **模块解耦**：各模块独立，易于维护

## 🎯 总结

Lobster系统的核心创新：

1. **群聊式交互** - Agent像真实团队成员一样协作
2. **主动感知决策** - Agent智能判断何时参与
3. **@提及召唤** - 直观的用户交互方式
4. **共享工作空间** - 所有成员看到相同的项目状态
5. **共享记忆** - 无需重复解释上下文
6. **灵活可扩展** - 支持动态添加任意数量的Agent

这是一个**真正智能的多Agent协作系统**，而不仅仅是自动化流水线！
