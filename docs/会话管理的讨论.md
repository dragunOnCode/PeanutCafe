## MODEL ✨

这是一个非常深刻的工程实践问题。你正试图在**“利用 CLI 工具的便捷性”**与**“实现复杂的 Agent 编排逻辑”**之间寻找平衡。

以下是对你思考过程的详细解答以及关于开源多 Agent 系统的调研：

### 1. 思考过程问题解答

#### (1) 关于成本：CLI vs API

**这是一个常见的误区。**

- **实质相同：** `claude code`、`gemini cli` 等工具本质上也是调用后端的 API。除非你使用的是某些带有固定月费、不计 Token 的特殊订阅版，否则它们产生的 Token 消耗最终都会反映在你的账单上（或消耗免费额度）。
- **CLI 的额外开销：** CLI 工具为了实现“自动搜索文件”、“阅读上下文”等功能，往往会在系统提示词（System Prompt）中注入大量的指令和当前工程的结构信息。这意味着**直接使用 CLI 往往比精简的 API 调用更贵**，因为 CLI 每次都会带上大量的“背景噪音” Token。

#### (2) & (3) 代理间对话与 Session Resume

- **状态隔离：** 你的观察是正确的。每个 CLI 进程都是一个独立的“孤岛”。
- **Resume 的局限：** `resume` 参数的作用是恢复该 Agent **之前与你的**对话历史。它无法感知并行发生的、其他 Agent 的对话。
- **解决方案：** 在多 Agent 架构中，通常需要一个**“黑板（Blackboard）”**或**“协调者（Orchestrator）”**。你需要将 Agent A 的输出提取出来，作为“上下文片段”喂给 Agent B。

#### (4) 历史记录 + Prompt 是否会污染上下文？

- **干扰风险：** 如果你不断地把“全量历史”通过 `prompt` 传给一个已经 `resume` 的会话，确实会造成**上下文污染**。模型会看到两份高度相似但结构略有不同的历史（一份是它内部 Session 记录的，一份是你手动输入的），这会导致：
  1.  **Token 浪费：** 重复计算相同的上下文。
  2.  **注意力分散：** 模型可能会混淆“谁在什么时候说了什么”。
- **最佳实践：**
  - 如果使用 `resume`，只发送**增量信息**（即其他 Agent 新产生的回复）。
  - 如果不使用 `resume`，则发送**格式化后的全量历史**。

#### (5) 舍弃 Resume 后的性能与空间影响

- **功能影响：** 大量的一次性会话（One-off sessions）通常不会影响 CLI 的运行效率。CLI 存储 Session 主要是为了方便用户回溯，通常是本地的 SQLite 或 JSON 文件。
- **检索效率：** 除非你一次性生成了数万个 Session 且该 CLI 带有全局索引功能，否则对性能影响微乎其微。
- **关键点：** 如果你选择每次携带全量历史，**你实际上是在自己管理 Context Window**。这比依赖 CLI 的 `resume` 更灵活，但也要求你必须处理好 Token 溢出（超过模型最大长度）的问题。

### 2. 开源多 Agent 协作系统及其工程实践

目前主流的开源系统并不是直接调用 CLI，而是通过封装 API 来实现复杂的协同逻辑。以下是代表性的系统：

#### A. Microsoft AutoGen (目前最流行)

- **实践方式：** 定义“可对话的代理（Conversable Agents）”。
- **核心机制：**
  - **消息总线：** Agent 之间通过标准化的 JSON 消息进行通信。
  - **自动编排：** 引入一个 `GroupChatManager`。当用户发帖后，Manager 会根据当前状态决定下一个由谁发言。
  - **上下文管理：** Manager 负责收集所有人的发言，并在轮到某个 Agent 时，将**经过剪裁和格式化**的历史记录发送给该 Agent。

#### B. CrewAI

- **实践方式：** 角色扮演（Role-Based）与流程驱动（Process-Driven）。
- **核心机制：**
  - **任务链：** 任务（Task）在 Agent 之间按顺序（Sequential）或等级（Hierarchical）传递。
  - **上下文注入：** 明确定义一个任务的 `context` 来源于前序哪几个任务的输出。它不会把所有废话都传下去，只传“交付物”。

#### C. MetaGPT

- **实践方式：** 将软件工程的 SOP（标准作业程序）引入 Agent。
- **核心机制：**
  - **共享白板：** 所有 Agent 都能看到一个“环境（Environment）”。
  - **发布/订阅模式：** Agent A 完成设计文档后，发布一个消息；订阅了“设计文档”事件的 Agent B（审核者）会自动触发工作。

### 3. 给你的建议方案

如果你坚持使用 CLI 工具来构建，建议采用以下架构：

1. **外部状态管理器：** 使用一个本地文件（如 `chat_history.json`）记录所有 Agent 的对话。

2. **不使用 `resume`：** 每次调用 CLI 时，从 `chat_history.json` 中提取最近的 $N$ 条记录，构造一个类似：

   ```text
   以下是目前的协作历史：
   [Codex]: 设计了数据库表结构...
   [Gemini]: 提出了索引优化建议...

   现在的任务是：[请你执行审核]
   ```

3. **解析输出：** 编写脚本捕获 CLI 的标准输出（Stdout），提取核心回复，更新到 `chat_history.json`。

4. **按需注入：** 不要给每个 Agent 发送全量历史。比如给“代码实现 Agent”发送“设计方案”即可，无需发送“关于 UI 颜色的讨论历史”。这样既能省钱，又能保持模型的高效。

**总结：** CLI 适合作为个人生产力工具，但作为系统的后端组件时，它的“黑盒” Session 管理反而会成为阻碍。**自建上下文管理 + 无状态 CLI 调用**是更稳健的路径。

## MODEL ✨

这是一个非常务实且具有工程前瞻性的思考。既然你决定利用 CLI 的“端到端执行能力（Agentic Capabilities）”和现有的订阅，那么**上下文管理（Context Management）将是你系统的核心挑战。**

直接回答你的问题：**是的，如果你不使用 `resume`，或者即使你使用 `resume` 但涉及多 Agent 协作，你都必须进行额外的上下文管理。**

以下是深度分析及建议的工程方案：

### 1. 为什么必须做上下文管理？

如果你每次都把“全量历史 + 当前 Prompt”发给 CLI：

- **Token 爆炸：** 随着对话轮数增加，输入 Token 数呈线性（甚至由于重复而呈指数级）增长。即使模型支持 1M 上下文，处理 50k+ Token 的首字延迟（TTFT）会显著增加。
- **注意力稀释（Lost in the Middle）：** 模型对长文本中间内容的理解力会下降。如果你把 Agent A 的废话、Agent B 的确认信息全部塞进去，Agent C 可能会忽略最核心的任务指令。
- **CLI 的特殊性：** 像 `claude code` 这种工具，它在启动时会扫描文件系统、读取环境信息。如果你再手动塞入大量的历史文本，会干扰它对当前工作目录（CWD）状态的判断。

### 2. 核心架构建议：从“对话历史”转向“状态同步”

既然 CLI 具有“干活”的能力（读写文件），你其实不需要把“怎么干的过程”全部传给下一个 Agent，而应该传递**“干完的结果”**。

我们可以借鉴分布式系统中的 **“共享存储（Shared Disk）”** 模式：

#### 方案 A：以文件系统作为“真理来源” (推荐)

与其传全量对话历史，不如让 Agent 互相对齐“工程状态”。

- **做法：**
  1.  Agent A 执行完命令后，会在本地生成/修改文件。
  2.  调用 Agent B 时，不再传 Agent A 说过的话，而是告诉 Agent B：`“Agent A 已经根据需求修改了 src/auth.js，请你基于该文件编写对应的单元测试。”`
  3.  CLI 工具（如 `claude code`）会自动通过 `ls` 或 `cat` 去看那个文件。
- **优点：** 极大地节省 Token，且 Agent 看到的是最新的代码，而不是“对话里描述的代码”。

#### 方案 B：维护一个“架构说明书（Architecture Decision Record, ADR）”

- **做法：** 创建一个隐藏文件（如 `.agent_context.md`）。
- **流程：**
  1.  每次 Agent A 完成一个阶段性任务，强制要求它更新这个 `.agent_context.md`。
  2.  调用下一个 Agent 时，Prompt 始终包含：`“请先阅读 .agent_context.md 了解当前项目进度和决策逻辑，然后执行任务 X。”`
- **效果：** 这本质上是人工实现的 **“状态压缩”**。

### 3. 如何处理必须要保留的“对话历史”？

如果某些业务逻辑只存在于对话中，没有体现在代码里，你必须管理历史，建议采取以下策略：

1.  **滑动窗口（Sliding Window）：** 只保留最近 5-10 轮的对话。
2.  **总结摘要（Summarization）：**
    - 当历史达到一定长度（如 4000 Token）时，调用一个廉价模型（甚至就是当前 CLI）执行：`“请总结以上对话的核心结论、已确认的需求和待办事项。”`
    - 后续调用时，只带上这个 **[总结] + [最近 2 轮对话] + [当前 Prompt]**。
3.  **元数据提取：**
    - 记录每个 Agent 的身份和贡献。例如：`[System Info]: Codex updated DB schema; Gemini approved safety protocols.` 这种简短的元数据比原始对话高效得多。

### 4. 关于 CLI 订阅与 API 限制的工程提醒

- **速率限制（Rate Limiting）：** 即使你有订阅，CLI 背后通常也有每分钟请求数（RPM）或每分钟 Token 数（TPM）的限制。多 Agent 并行或频繁调用很容易触发 `429 Too Many Requests`。你需要在你的调用代码中加入 **指数退避重试（Exponential Backoff）**。
- **并发冲突：** 如果 Agent A 正在修改 `file.ts`，Agent B 同时也试图修改它，CLI 的文件锁或模型对过时代码的读取会导致逻辑错误。你需要一个简单的 **任务队列（Task Queue）** 来确保对本地文件系统的修改是串行的（或者通过 Git 分支隔离）。

### 总结建议

1.  **放弃“同步全量对话”的执念。** 你的 Agent 不是在写小说，而是在协同开发。
2.  **利用“环境即上下文”。** 既然用了 CLI，就让 Agent 更多地通过 `ls`, `grep`, `cat` 来获取信息，而不是靠你通过 Prompt 喂给它。
3.  **实现一个简易的“Orchestrator”（调度器）。** 这个脚本负责：
    - 记录全局状态。
    - 分发任务。
    - 定期清理/总结冗余信息。
    - 处理各 CLI 之间的输出解析。

这样做，你就能在享受 CLI 订阅和强大工具能力的同时，避开 Token 溢出的陷阱。
