# 上下文管理模式现状与实施计划

## 1. 背景与目标

基于当前决策：

- 去除 `gemini.adapter.ts`、`claude.adapter.ts`、`codex.adapter.ts` 中的 CLI 续聊参数（如 `-r` / `resume`）。
- 会话上下文统一由后端系统内部管理，不再依赖各 CLI 的内建会话状态。

本文档目标是给出：

1. 当前系统上下文管理能力的真实现状；
2. 与讨论中“多模式上下文管理”方案的差距；
3. 可执行的分阶段实现计划（先可用、再优化、再稳定）。

## 2. 当前系统现状梳理（As-Is）

## 2.1 消息与记忆数据流

当前主链路（`ChatGateway -> ChatService -> AgentAdapter`）如下：

1. 用户消息进入 `ChatGateway`，经 `messageRouter` 归一化后保存；
2. `ChatService.saveMessage` 将消息写入：
   - 短期记忆（Redis，`ShortTermMemoryService`）；
   - 向量库（Chroma，`messages` 集合）；
   - 可选持久化 DB / 进程内缓存；
3. Agent 执行前，`ChatGateway` 会拉取最近 20 条消息构造 `AgentContext.conversationHistory`；
4. Agent 返回后再次落消息，形成下一轮可见历史。

结论：系统已具备“消息级历史沉淀 + 向量索引 + Redis 短期窗口”的基础能力。

## 2.2 ContextBuilder 与语义检索现状

`ContextBuilderService` 已可构建增强上下文：

- 语义检索（`messages` 集合）；
- 摘要检索（`summaries` 集合）；
- Redis 短期记忆回退；
- SharedMemory 工作区状态注入。

但当前实际生效范围不一致：

- **ClaudeAdapter**：会调用 `contextBuilder.buildContext(...)`，并执行历史裁剪、去重、token 预算控制；
- **CodexAdapter / GeminiAdapter**：未接入 `ContextBuilder`，基本直接把 prompt 交给 CLI。

结论：上下文构建能力存在，但仅 Claude 使用，系统层尚未统一。

## 2.3 SharedMemory 现状

`SharedMemoryService` 当前存储三类信息：

- `workspaceState`：最后用户消息等工作区状态；
- `decision`：agent 是否响应与原因；
- `agentThreadBinding`：业务 session 与 CLI thread/session 的映射（当前用于续聊）。

结论：共享内存已存在且仍是必需能力（尤其 `workspaceState` 与 `decision`）；仅 `agentThreadBinding` 与“去续聊”目标冲突，需要逐步下线。

## 2.4 Adapter 续聊耦合现状

三个 CLI Adapter 当前都保留续聊逻辑：

- `ClaudeAdapter`：存在 session 绑定时使用 `-r`；
- `GeminiAdapter`：存在映射时使用 `-r`；
- `CodexAdapter`：存在 thread 绑定时使用 `exec resume`。

并带有“续聊失败 fallback 到新会话”的分支。

结论：当前上下文策略是“后端历史 + CLI 内建续聊”混合模式，存在状态重复与语义冲突风险。

## 2.5 已有能力与缺口总结

### 已有能力

- 消息沉淀、短期记忆、向量检索链路齐备；
- Claude 已有上下文裁剪与预算控制实践；
- 同一轮中后执行 Agent 可看到前序 Agent 最新回复（串行执行 + 每次刷新历史）。

### 关键缺口

- 上下文构建未统一到所有 Agent；
- 缺少系统级“上下文策略层”（不同 Agent、不同任务的注入规则）；
- `summaries` 集合有检索能力但缺生成与维护流程；
- 仍依赖 CLI 续聊映射，违背“后端统一管理上下文”目标。

## 3. 目标架构（To-Be）

## 3.1 设计原则

1. **单一事实来源**：后端上下文为唯一可信状态，CLI 仅做无状态执行器；
2. **统一上下文封装**：所有 Adapter 接收同构 `AgentContext`；
3. **分层注入**：短期窗口 + 语义记忆 + 摘要记忆 + 工作区状态；
4. **预算可控**：按 Agent 类型执行 token/字符预算与裁剪；
5. **可观测**：每次调用记录上下文规模、命中来源、裁剪结果。

## 3.2 上下文管理模式落地（对应讨论方案）

### 模式 A：环境即上下文（文件系统/工作区状态）

- 保留并增强 `workspaceState`；
- 在 prompt 协议中明确“优先读取当前工作区事实，不依赖历史转述”；
- 补充关键改动摘要（最近变更文件、最近决策）。

### 模式 B：压缩状态文档（ADR/会话摘要）

- 增加会话级压缩载体（建议 `session summary + decisions`）；
- 将阶段性结论写入 `summaries` 向量集合；
- 每次调用注入“最近摘要 + 当前问题”，替代全量历史。

### 模式 C：滑动窗口 + 语义召回

- 统一为所有 Agent 提供：
  - 最近 N 条消息窗口；
  - 语义 Top-K 历史片段；
  - token/字符预算截断。
- 超预算时优先保留：当前问题 > 最近轮次 > 高相似语义片段 > 低价值寒暄。

### 模式 D：结构化元数据同步

- 继续维护并扩展：
  - `workspaceState`
  - `decision`
  - `lastAgentOutputs`（新增，保存每个 Agent 最近关键结论）
- 用结构化字段替代冗长自然语言历史传递。

## 4. 分阶段实施计划

## Phase 0：冻结目标与兼容策略（0.5 天）

目标：先对齐边界，避免改造中断主链路。

- 明确“禁用续聊”的发布策略：一次切换或灰度开关；
- 明确各 Agent 的默认上下文预算；
- 确认回滚方案（保留旧分支，不在主干残留双逻辑过久）。

交付物：

- 本文档评审通过；
- 变更清单与验收标准确认。

## Phase 1：去除 Adapter 续聊参数（1 天）

目标：让 CLI 调用完全无状态。

- `ClaudeAdapter` 移除 `resolveExistingSessionId/persistSessionBinding/runClaude(...existingSessionId)` 相关续聊分支；
- `GeminiAdapter` 移除 `-r` 续聊分支与 session 映射缓存；
- `CodexAdapter` 移除 `exec resume` 分支与 thread 映射缓存；
- 清理 `SharedMemoryService` 的 `agentThreadBinding` 读写调用；
- 同步更新 adapter 单测（删除“应使用 -r/resume”断言）。

验收标准：

- 三个 Adapter 不再生成任何续聊参数；
- 相关测试通过，且无 thread binding 依赖。

## Phase 2：统一上下文构建入口（1~2 天）

目标：Claude/Codex/Gemini 使用同一套后端上下文。

- 在 `ChatGateway` 或新建 `ContextOrchestratorService` 中统一构建 `AgentContext`；
- `CodexAdapter`、`GeminiAdapter` 接入 `ContextBuilderService` 产物；
- 统一 prompt 协议模板（当前问题、对话窗口、语义参考、摘要参考、工作区状态）；
- 提供按 Agent 定制的裁剪参数（窗口大小、语义条数、摘要条数、最大字符数）。

验收标准：

- 三个 Agent 均能读取统一结构化上下文；
- 同一输入在三 Agent 路径上可观测到上下文注入日志。

## Phase 3：补齐摘要管理模式（2 天）

目标：打通“摘要生成 -> 存储 -> 召回 -> 注入”闭环。

- 增加 `ConversationSummaryService`：
  - 按消息数或 token 阈值触发摘要；
  - 生成结构化摘要（已确认需求/关键决策/待办/风险）；
- 摘要写入 `summaries` 向量集合；
- `ContextBuilderService` 优先拉取近期摘要并注入；
- 增加摘要去重与版本化策略（避免重复写入）。

验收标准：

- `summaries` 集合有持续新增数据；
- 长会话中上下文长度增长受控，响应质量稳定。

## Phase 4：观测与质量兜底（1~2 天）

目标：可量化评估新模式效果并快速排障。

- 增加上下文观测字段：
  - `contextChars/contextEstimatedTokens`
  - `historyItems/semanticItems/summaryItems`
  - `trimmedItems`
  - `buildContextLatencyMs`
- 建立回归测试：
  - 长会话（>50 轮）；
  - 多 Agent 串行协作；
  - 语义检索不可用降级场景；
- 增加失败保护：上下文构建异常时退化为最近窗口消息。

验收标准：

- 能通过日志快速定位“上下文过大/召回无效/注入缺失”问题；
- 核心链路在依赖降级时仍可用。

## 6. 风险与应对

1. **上下文过长导致超时**
   - 应对：统一预算裁剪 + 失败退化到最近窗口。
2. **语义召回噪声高**
   - 应对：提高 `minSimilarity`、增加角色过滤、摘要优先。
3. **改造期行为回归**
   - 应对：按 Phase 分步上线，保留关键集成测试。
4. **多 Agent 信息不一致**
   - 应对：统一构建入口，禁止 Adapter 私有上下文拼接策略漂移。

---

## 7. 本阶段结论

当前系统已具备“后端管理上下文”的基础设施，但实现上仍是混合模式（后端 + CLI 续聊）。下一步应优先完成“去续聊 + 上下文统一构建”，随后补齐摘要闭环和观测能力。完成后可形成稳定的多 Agent 协作上下文管理体系，并显著降低状态重复与上下文污染风险。
