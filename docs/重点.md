设计一个足以支撑高级技术面试（System Design 或 AI Engineering）的 Agent 系统，不能仅停留在“调通 API”的层面，而需要展示**架构深度、工程化落地能力以及边缘情况的处理**。

以下是构建生产级 Agent 系统必须考虑的六大核心维度：

### 1. 核心架构与推理模式 (Architecture & Reasoning)

面试官关注你是否理解 Agent 并非单纯的 `Ask -> Answer`，而是循环的决策过程。

- **认知架构 (Cognitive Architecture):**
- **ReAct (Reason + Act):** 最经典的模式。展示你如何让 Agent 在“思考-行动-观察”的循环中工作。
  _ **Planning (规划):** 对于复杂任务，Agent 是否有能力进行任务拆解 (Decomposition)？例如使用 Chain of Thought (CoT) 或 Tree of Thoughts (ToT)。
  _ **多 Agent 协作 (Multi-Agent):** 是否涉及像 MetaGPT 或 AutoGen 那样的角色分工（如 Manager, Coder, Reviewer）来提升复杂任务的成功率。

### 2. 记忆管理 (Memory Management)

LLM 是无状态的，如何处理长期和短期记忆是系统设计的关键。

- **短期记忆 (Short-term):**
- **Context Window 限制:** 如何处理超长上下文？策略包括：滑动窗口 (Sliding Window)、摘要 (Summarization)、关键信息提取。
- **长期记忆 (Long-term):**
- **RAG (检索增强生成):** 使用向量数据库 (Vector DB) 存储历史交互或知识库。
  \_ **记忆分层:** 区分“事实性记忆”（用户叫什么名字）和“过程性记忆”（上次任务是如何解决的）。

### 3. 工具使用与交互 (Tool Use / Function Calling)

这是 Agent 的“手”，决定了它能做什么。
\_ **接口定义:** 如何设计清晰的 Schema (OpenAPI/JSON Schema) 让模型准确理解工具用途。

- **容错与自我修正:** 如果工具调用失败（如 API 500 或参数错误），Agent 能否根据错误信息自动重试或修正参数，而不是直接崩溃？
- **人机交互 (HITL - Human in the Loop):** 关键操作（如转账、删除文件）是否设计了人工确认机制。

### 4. 评估与观测 (Evaluation & Observability)

这是区分 Demo 和生产级系统的分水岭。

- **评估框架:** 如何衡量 Agent 的好坏？
- **成功率 (Success Rate):** 任务是否完成？
  _ **步骤效率:** 完成任务花了多少步？有没有死循环？
  _ **准确性:** 是否有幻觉 (Hallucination)？
- **调试工具:** 是否集成了像 LangSmith, LangFuse 或 Arize Phoenix 这样的工具来追踪 Trace，分析每一步的 Token 消耗和耗时。

### 5. 工程化挑战 (Engineering Challenges)

- **延迟优化 (Latency):** Streaming (流式输出) 是标配。如何通过并行调用 (Parallel Function Calling) 减少等待时间？
- **成本控制 (Cost):** 如何在 GPT-4 (用于推理) 和 GPT-3.5/Haiku (用于总结/简单任务) 之间进行模型路由 (Model Routing)。
- **状态管理:** 如果服务重启，Agent 当前执行到一半的任务状态（State）能否恢复？

### 6. 安全与护栏 (Safety & Guardrails)

- **Prompt Injection:** 防止用户诱导 Agent 执行违规操作。
- **数据隐私:** 敏感数据（PII）在发送给 LLM 前是否进行了脱敏。\* **输出校验:** 确保输出格式符合预期（如强制 JSON 格式），防止下游服务崩溃。

### 总结：面试中的“加分项”

如果你能具体阐述以下场景的解决方案，会非常有说服力：

1. **“当 Agent 陷入死循环时，系统如何检测并打破循环？”**
2. **“如何解决工具参数生成的幻觉问题（调用了不存在的参数）？”**
3. **“如何设计一套自动化的 Evals 流水线来测试 Prompt 的迭代？”**
