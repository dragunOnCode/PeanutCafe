## 实施策略

### 核心原则（遵循KISS）

1. **先实现核心流程**：用户发消息 → Agent响应 → 保存结果
2. **再优化性能**：缓存、异步、队列
3. **最后增强功能**：监控、分析、高级特性

### 从OpenClaw学到的教训

根据OpenClaw实践经验，**90%的失败来自记忆管理**：
- ❌ 无限制存储所有对话 → 上下文爆炸
- ❌ 无记忆摘要 → Agent忘记重要信息
- ❌ 无结构化存储 → 检索效率低

**Lobster的解决方案**：
- ✅ 三层记忆架构（短期/长期/共享）
- ✅ 自动摘要（每小时触发）
- ✅ 向量检索（语义搜索）
- ✅ 文件系统审计日志

## 技术风险评估

### 高风险项

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| **CLI调用失败** | Agent无法响应 | 1. 增加重试机制<br>2. 准备降级方案<br>3. 详细错误日志 |
| **记忆爆炸** | 上下文过长、性能下降 | 1. 自动摘要（每小时）<br>2. 滑动窗口（最近N条）<br>3. 向量检索（语义过滤） |
| **并发竞态** | 多Agent同时写文件 | 1. 文件锁机制<br>2. 版本号控制<br>3. 乐观锁 |
| **WebSocket断线** | 消息丢失 | 1. 自动重连<br>2. 消息队列缓冲<br>3. 离线消息推送 |

### 中风险项

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| **Agent响应慢** | 用户体验差 | 1. 流式响应<br>2. 超时提示<br>3. 取消机制 |
| **存储空间** | 磁盘占满 | 1. 定期清理<br>2. 归档到云存储<br>3. 压缩历史数据 |
| **Redis内存** | OOM | 1. 设置合理TTL<br>2. 淘汰策略<br>3. 监控告警 |

## 快速启动方案

### 30分钟MVP（最小可运行版本）

**目标**：验证核心流程可行性

**实现内容**：
1. ✅ NestJS项目 + Socket.io
2. ✅ 一个Agent（Claude）
3. ✅ 内存存储（不持久化）
4. ✅ 简单的消息收发

**快速代码**：

```typescript
// mini-mvp/main.ts
import { NestFactory } from '@nestjs/core';
import { IoAdapter } from '@nestjs/platform-socket.io';

async function bootstrap() {
  const app = await NestFactory.create(AppModule);
  app.useWebSocketAdapter(new IoAdapter(app));
  await app.listen(3000);
}

// mini-mvp/chat.gateway.ts
@WebSocketGateway()
export class ChatGateway {
  @WebSocketServer()
  server: Server;
  
  private claudeAdapter = new ClaudeAdapter();
  
  @SubscribeMessage('message')
  async handleMessage(@MessageBody() data: string) {
    // 1. 调用Claude
    const response = await this.claudeAdapter.generate(data, {});
    
    // 2. 推送响应
    this.server.emit('response', {
      agent: 'Claude',
      content: response.content
    });
  }
}
```

**测试**：
```bash
npm install
npm run start:dev

# 测试WebSocket连接
node test-client.js
```

## 具体实现指南

### 实现顺序建议

**阶段1：垂直切片（端到端）**

实现一个完整的用户故事：

```
用户发送"Hello" → Claude响应"Hi!" → 前端显示
```

包含：
- ✅ Gateway接收消息
- ✅ Agent生成响应
- ✅ WebSocket推送
- ✅ 数据库保存

**代码示例**（完整流程）：

```typescript
// 1. Gateway接收消息
@SubscribeMessage('message:send')
async handleMessage(client: Socket, data: { content: string; sessionId: string }) {
  // 2. 保存到数据库
  const message = await this.messageRepo.save({
    sessionId: data.sessionId,
    role: 'user',
    content: data.content
  });
  
  // 3. 广播给房间
  this.server.to(`session:${data.sessionId}`).emit('message:received', message);
  
  // 4. 调用Agent
  const agent = this.agentService.getAgent('claude-001');
  const response = await agent.generate(data.content, {});
  
  // 5. 保存Agent响应
  const agentMessage = await this.messageRepo.save({
    sessionId: data.sessionId,
    role: 'assistant',
    agentId: agent.id,
    content: response.content
  });
  
  // 6. 推送给前端
  this.server.to(`session:${data.sessionId}`).emit('message:received', agentMessage);
}
```

## 总结

### OpenClaw架构的核心价值

1. **Hub-and-Spoke简化架构** → Lobster完全采用
2. **文件系统工作空间** → Lobster增强采用（+云存储）
3. **Config热重载** → Lobster完全采用
4. **Transcript审计日志** → Lobster完全采用（JSONL）
5. **插件化扩展** → Lobster完全采用
6. **Cron定时任务** → Lobster采用（使用BullMQ）

### Lobster的差异化创新

1. **多Agent群聊式交互** → 区别于OpenClaw的一对一
2. **Agent主动感知决策** → 更智能的响应机制
3. **共享工作空间** → 所有Agent协作而非隔离
4. **实时状态同步** → Redis + Pub/Sub
5. **双模式切换** → 自由对话 + 结构化工作流

### 最终架构

**Lobster = OpenClaw的骨架 + 群聊式交互创新**

- ✅ 借鉴：Gateway架构、文件系统、热重载、日志审计
- ✅ 创新：多Agent协作、主动决策、共享记忆
- ✅ 简化：去除多Channel、mDNS等非必需组件
- ✅ 增强：实时缓存、向量检索、工作流编排
